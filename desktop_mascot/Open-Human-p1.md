## 1. High‑level architecture you’ll build

Think of three big parts:

1. **Vyomanta LMS front‑end** (what you already have)
    - Next.js web app + React UI.[^1]
    - PersonalizedBot.jsx overlay, labs, tutor views.[^1]
2. **Desktop mascot app**
    - Transparent always‑on‑top window with your 3D bot.[^1]
    - Could stay PyQt + Three.js or later move to something else.
3. **Local “OpenHuman‑like” agent** (your scratch implementation)
    - Runs as a separate process on the user’s machine.
    - Has:
        - **Memory store** over user events.
        - **Skill system** (small tools you call: “read LMS state”, “open route”, “suggest next step”).
        - **Planner/LLM layer** that decides what to do.[^2][^3]

Data flow:

- LMS → Agent: “Here is the user’s current state (page, code, errors, etc.). What hint/action/animation state should the bot take?”[^1]
- Agent → LMS + Desktop: “Play `THINKING` animation, show hint, maybe navigate to /playground.”[^4][^1]

## 2. Build the local agent from scratch

### 2.1. Process + API layer

Terms:

- **Local agent process** – a long‑running backend program on the user’s machine.
- **RPC / API** – how your LMS and desktop app talk to it (HTTP or WebSocket).

Steps:

1. Create a small **Node.js or Python service** that runs locally (e.g., on `localhost:7000`).
2. Define a simple JSON API like:
    - `POST /ingest/context` – LMS sends current page, code, errors, timestamps.[^1]
    - `POST /agent/decide` – LMS or desktop asks: “Given current context, what should the tutor do?”
    - `GET /memory/query` – later, you can query what the user did before.
3. Wire your **PersonalizedBot.jsx** to call `/ingest/context` whenever page or code changes, and `/agent/decide` whenever the user asks for help.[^1]

This is your “Option A agent” in minimal form.

### 2.2. Memory system (your NeoCortex‑lite)

Terms:

- **Event log** – append‑only list of user events (visited routes, compile errors, solved tasks).
- **Memory graph / tree** – later, a structured representation; initially, just indexed JSON.[^5][^2]

Steps:

1. Start with a **SQLite or simple JSON file** storing events:
    - `timestamp`, `route`, `action`, `code_snapshot`, `error_message`, `quiz_score`, etc.
2. Add a **“ingest” module** in the agent that:
    - Normalizes events (e.g., `type: "ERROR"`, `type: "PAGE_VIEW"`).
    - Saves them with user id (if multi‑user).
3. Add a small **retrieval function**:
    - `get_recent_errors_for_route(route)`
    - `get_common_pages_after_error(error_type)`

This gives you “long‑term memory over the user’s digital learning life” in a simple way, similar in spirit to OpenHuman but much lighter.[^2][^1]

### 2.3. Skill system

Terms:

- **Skill** – a small function the agent can call, like “navigate to route”, “generate hint”, “ask quiz”.[^3][^6]
- **Skill registry** – list of all skills + metadata.

Steps:

1. Design skills as **plain functions** in your agent:
    - `skillSuggestHint(context)` – returns Socratic questions instead of full answers.[^1]
    - `skillNavigate(route)` – encodes `[ACTION: navigate, route:'/playground']` for the LMS to execute.[^1]
    - `skillSetAnimationState(state)` – tells the mascot which animation to play.
2. Keep a registry:

```json
[
  { "name": "suggest_hint", "input": "LMSContext", "output": "TutorMessage" },
  { "name": "navigate", "input": "Route", "output": "ActionTag" },
  { "name": "set_animation", "input": "BotState", "output": "AnimationCommand" }
]
```

3. Your agent’s “planner” chooses which skill to execute based on context and optionally an LLM.[^3][^2]

## 3. Animation + “antigravity” from scratch (terms you can use)

You want professional‑style floating motion. Here are the key terms and components you can use in prompts and design.

### 3.1. Animation state machine

Terms:

- **Animation state** – named modes like `IDLE_FLOAT`, `FOCUS`, `TALKING`, `ERROR_REACT`.[^1]
- **State transition** – rule that switches from one state to another (e.g., `IDLE_FLOAT → TALKING` when TTS starts).
- **Blend / easing** – smooth interpolation between states.

Define states:

- `IDLE_FLOAT` – default, bot gently hovering (“antigravity”).
- `FOCUS` – when user is reading or typing; bot leans forward slightly.
- `TALKING` – when your TTS or voice tutor is active.[^1]
- `SURPRISE_ERROR` – quick bounce/shake when a compile error appears.[^1]

Your agent’s skill `set_animation_state` decides which of these is active.

### 3.2. Antigravity motion system (what to ask for in a prompt)

For an LLM or animation tool, you can describe it like this:

- **Hover / bobbing**: Y‑position moves as a **sinusoidal function of time**:
    - Term: “sine‑wave vertical oscillation” or “procedural hover animation”.
- **Micro‑rotation**: small rotation around Y and Z:
    - Term: “subtle tilt around yaw and roll axes”.
- **Noise**: add small random offsets to amplitude and phase:
    - Term: “Perlin noise modulation for organic motion” (more advanced) or just “random jitter”.
- **Easing**: when changing states, positions and rotations use “smooth easing curves” (ease‑in‑out).

So a clean prompt describing your antigravity:

> “Create a procedural antigravity idle animation for my 3D robot mascot: a sine‑wave vertical hover, subtle yaw and roll tilts, and low‑amplitude Perlin noise on position and rotation, with smooth ease‑in‑out transitions between states like IDLE_FLOAT, TALKING, and SURPRISE_ERROR.”

These are the **terms** you can reuse when prompting animation tools or LLMs.

## 4. Connecting the LMS + agent + animation

Now tie everything together:

### 4.1. LMS → agent

From `PersonalizedBot.jsx`, send context:

- `route`, `activeComponent`, `code`, `stderr`, `userAction` (typing, scrolling).[^1]

Call:

- `POST /agent/decide` with that context.

Agent returns:

- `message` (Socratic hint).
- `action` (e.g., `{ type: "navigate", route: "/playground" }`).[^1]
- `animationState` (e.g., `"TALKING"`).


### 4.2. Agent → desktop mascot

Desktop app (PyQt or your future stack) listens on WebSocket:

- When it receives `{ animationState: "IDLE_FLOAT" }`, it applies antigravity idle motion.
- When `{ animationState: "TALKING" }`, it plays a version with slightly larger hover amplitude and maybe head motion.

Animation system:

- Core concept: **procedural animation controller** that maps `animationState` + `time` → transforms (position/rotation/scale).
